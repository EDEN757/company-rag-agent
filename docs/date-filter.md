# Known limitation: the date filter is effectively gmail-only

## TL;DR

The `date_from` / `date_to` parameters of the `search` tool only constrain
**gmail** chunks. Slack, Confluence, Jira, Linear, HubSpot, GitHub, Fireflies
and Google Drive chunks pass through the filter unchanged, regardless of when
they were created.

This is documented behavior, not a bug to ignore — but it does mean queries
like "what was discussed in November 2026" will return all non-gmail content
in the corpus *plus* the gmail chunks that actually fall in the window.

## Why this happens

The `chunks` table has two timestamp columns, `ts_from` and `ts_to`, populated
during chunking (`indexing/chunkers.py`). Only `chunk_gmail` parses the
`Date:` header of each email and writes those columns. Every other source
type goes through `chunk_document_like` or `chunk_slack`, both of which
leave the timestamps `NULL`.

The SQL filter generated by `src/rag/fusion.ts:buildFilterClause` is:

```sql
WHERE (ts_to   IS NULL OR ts_to   >= :date_from)   -- when date_from given
  AND (ts_from IS NULL OR ts_from <= :date_to)     -- when date_to given
```

For any chunk with both timestamps `NULL`, both clauses are satisfied via the
`IS NULL` branch, so the row passes through. This was a deliberate "treat
unknown timestamps as could-match-anything" choice — sensible in isolation,
misleading in aggregate because most of the corpus has unknown timestamps.

## What this means in practice

| User asks                                | What they probably expect             | What they get                                                |
| ---------------------------------------- | ------------------------------------- | ------------------------------------------------------------ |
| "Emails from October about the budget"   | only gmail from October               | ✅ correct — gmail is fully filtered                          |
| "What was discussed in Slack last week"  | slack from last week                  | ⚠️ all slack (date filter ignored on slack)                  |
| "November conversations across the org"  | gmail + slack from November           | ⚠️ gmail from November + all slack/confluence/jira/… ever    |

The agent's system prompt should bias toward applying date filters only when
the user explicitly references emails, until this limitation is lifted.

## Future fix (not done — see options below)

When/if we want a real cross-source date filter, there are two paths:

### Option A — populate timestamps in every chunker

Most source documents in the underlying parquet have timestamp metadata; the
chunkers just don't extract it. Slack messages carry per-message timestamps,
Jira/Linear/Confluence/HubSpot/GitHub all have `created_at` / `updated_at`.

Work involved:
1. Extend each chunker in `indexing/chunkers.py` to surface a `ts_from` /
   `ts_to` per chunk (min/max of contained messages for slack, doc-level
   timestamp for the rest).
2. Re-chunk and re-embed each affected source incrementally using
   `indexing/rechunk_source.py --source <name>`. Embeddings don't actually
   change (chunk text didn't), so a simpler script could just UPDATE the
   `chunks` table — but the existing tool is the safer path.

This is the "right" fix. Estimated ~1–3 h of work depending on how messy
the per-source metadata is.

### Option B — make NULL exclude instead of include

Change the SQL to `AND ts_to IS NOT NULL AND ts_to >= ?` so that null
timestamps are excluded when a date filter is active. Predictable, one-line
change, but trades coverage: the date filter now silently hides almost the
entire non-gmail index. Probably not what users want; only worth it if we
also do Option A.

### Option C (chosen for now) — document the limitation

What this file does. The search tool's parameter descriptions also call out
the caveat so the LLM sees it at every tool invocation. No code change to
the filter logic itself.
